# Backend — Architecture

> **Stack:** Python 3.12 · FastAPI · SQLAlchemy + Alembic · Celery/RQ · PostgreSQL · Redis · MinIO (S3-compatible)
>
> **Role in the platform:** REST API server + async job worker (two separate processes sharing the same codebase)

---

## Design Philosophy

The backend follows a **layered architecture** with strict separation of concerns:

- **Routes** (`app/routes/`) — HTTP I/O only. Validate, delegate, return. No business logic.
- **Repositories** (`app/repositories/`) — All DB access. No HTTP, no business logic.
- **Domain models** (`app/models/`) — Pure Python dataclasses. No ORM, no HTTP.
- **DB models** (`app/db/models/`) — SQLAlchemy ORM. Separate from domain to allow schema evolution.
- **Processors** (`app/processors/`) — Pure functions: take an input file path, return a result dict.
- **Core** (`app/core/`) — Cross-cutting concerns: settings, logging, notifications, metrics, storage.

## Request Lifecycle (API)

```
Browser (or frontend server)
    │
    ▼
POST /jobs   ← FastAPI route
    │
    ├── StorageClient.object_exists()   ← verify input file in MinIO
    ├── Job domain model constructed
    ├── repo.create_job()               ← INSERT to PostgreSQL
    ├── repo.mark_queued()              ← UPDATE status → QUEUED
    └── JobQueue.enqueue()              ← RPUSH to Redis list
    │
    ▼
201 Created { job_id, status: "QUEUED" }
```

## Worker Lifecycle

The worker runs as a **separate process/pod** from the API. It shares the same Python codebase but is started with a different entrypoint:

```
worker.run_worker()   ← infinite poll loop
    │
    ├── start_http_server(8000)     ← Prometheus metrics endpoint
    ├── queue.dequeue(timeout=5)   ← blocking BRPOP on Redis; wakes on new job
    ├── repo.claim_next_job()       ← SELECT + UPDATE status → PROCESSING (atomic)
    │
    ├── prepare_workspace(job_id)   ← mkdir /tmp/jobs/{job_id}/
    ├── fetch_input()               ← storage.download_file() from MinIO input bucket
    ├── execute_processor()         ← processor.process(payload) → result dict
    ├── persist_output()            ← write result.json → storage.upload_file() to MinIO output bucket
    │                                  key: outputs/{job_id}/result.json
    ├── mark_completed()            ← UPDATE status → COMPLETED, set output_file_path
    └── dispatcher.dispatch(SUCCESS) ← send email if configured
```

On any exception:
```
    ├── handle_failure()           ← UPDATE status → FAILED (or DEAD if retries exhausted)
    └── dispatcher.dispatch(FAILURE) ← send failure email if configured
```

## Processor Pattern

All processors implement `BaseProcessor`:

```python
class BaseProcessor(ABC):
    @abstractmethod
    def process(self, payload: dict) -> dict:
        """
        payload keys:
          job_id, job_type, input_file_path (local /tmp path), input_metadata
        Returns a result dict that will be JSON-serialized as output.
        """
```

The registry maps enum values to processor singletons:

```python
PROCESSORS = {
    JobType.TEST_JOB:          TestProcessor(),
    JobType.CSV_ROW_COUNT:     RowCountProcessor(),
    JobType.CSV_COLUMN_STATS:  ColumnStatsProcessor(),
    JobType.CSV_DEDUPLICATE:   DeduplicateProcessor(),
    JobType.JSON_CANONICALIZE: JsonCanonicalizeProcessor(),
}
```

**Adding a new processor:**
1. Create `app/processors/<category>/<name>.py` implementing `BaseProcessor`
2. Add the new `JobType` enum value to `app/core/enums/job_type.py`
3. Register it in `app/processors/registry.py`
4. Add `JOB_TYPE_LABELS` entry in `frontend/lib/api.ts`

## Storage Layer

`app/core/storage.py` wraps the MinIO Python SDK behind a clean interface:

```python
class StorageClient:
    def object_exists(bucket, object_key) -> bool
    def download_file(bucket, object_key, local_path)
    def upload_file(local_path, bucket, object_key, content_type)
    def list_objects(bucket, prefix) -> list[str]
    def delete_object(bucket, object_key)
```

The worker downloads input files to a temporary local workspace (`/tmp/jobs/{job_id}/`), processes them on disk, then uploads the result. Temporary files are left on disk (not cleaned up) — acceptable for a learning project but would need a cleanup job in production.

## Notification Architecture

Notifications are a **non-blocking side effect** — they never affect job execution outcome.

```
Worker → dispatcher.dispatch(job, JobEvent.SUCCESS)
              │
              └── MailtrapEmailProvider.notify(job, event)
                        │
                        ├── Check notifications.email.enabled
                        ├── Check event in notifications.email.on
                        ├── Check context.email is present
                        └── Send via Mailtrap SDK (sandbox, no real delivery)
```

If the provider raises any exception, it is caught and logged. The job remains `COMPLETED`.

**Current provider:** Mailtrap (sandbox mode only). Does **not** deliver to real inboxes. Suitable for dev/testing.

**Adding a new provider:** Implement the provider interface, register in `NotificationDispatcher.__init__`.

## Prometheus Metrics

The worker exposes metrics on port `8000` (scraped by Prometheus via `ServiceMonitor`):

| Metric                        | Type      | Labels               | Description          |
| ----------------------------- | --------- | -------------------- | -------------------- |
| `worker_jobs_total`           | Counter   | `job_type`, `status` | Total jobs processed |
| `worker_job_duration_seconds` | Histogram | `job_type`           | Time per job         |

The backend API also exposes metrics via `GET /metrics` (via `prometheus_fastapi_instrumentator` or a custom route in `app/core/metrics.py`).

## Configuration (Environment Variables)

All config is in `app/core/settings.py` via `pydantic-settings`. Variables can be set via:
- `.env` file (local dev)
- Kubernetes `ConfigMap` + `Secret` (production via Helm)

| Variable                              | Default                                                 | Notes                            |
| ------------------------------------- | ------------------------------------------------------- | -------------------------------- |
| `POSTGRES_USER/PASSWORD/DB/HOST/PORT` | `postgres/postgres/notifications/postgres-service/5432` | DB connection                    |
| `REDIS_HOST/PORT/DB`                  | `redis-service/6379/0`                                  | Queue broker                     |
| `S3_HOST/PORT/USE_SSL`                | `minio/9000/false`                                      | MinIO endpoint                   |
| `S3_ACCESS_KEY/SECRET_KEY`            | `minioadmin/minioadmin`                                 | MinIO credentials                |
| `S3_INPUT_BUCKET`                     | `resilient-async-job-processing-inputs`                 | Upload target                    |
| `S3_OUTPUT_BUCKET`                    | `resilient-async-job-processing-outputs`                | Result storage                   |
| `MAILTRAP_API_KEY`                    | `""`                                                    | Required for email notifications |
| `MAILTRAP_USE_SANDBOX`                | `true`                                                  | `false` for real email sending   |
| `MAILTRAP_INBOX_ID`                   | `""`                                                    | Sandbox inbox ID                 |

## Local Development

```bash
cd backend
cp .env .env.local   # adjust values if needed
docker compose up    # starts postgres + redis + minio
uv run uvicorn main:app --reload --port 5001
```

The worker runs separately:
```bash
uv run python -m app.workers.worker
```

Or use the full `docker compose up` which starts both API and worker containers.
